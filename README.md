# capability-specs

一个文件夹一个能力：沉淀 Spec 索引与实战笔记，让 AI 快速落地真实任务。

## 这个仓库在做什么
互联网的信息很深、很散。这个项目把高价值方法整理成可复用 Spec，目标是更快实现一件事，而不是只停留在概念层。

## 当前能力

### 结构约定

- `场景型能力组`：把一个真实协作流程拆成多个小能力，适合复杂任务分步执行
- `特化能力`：针对某个业务上下文沉淀的专用能力，适合在明确场景里直接复用

### 场景型能力组

- [`specs/GitHub仓库基础协作`](./specs/GitHub仓库基础协作)
  面向不熟悉 Git / GitHub 的执行者，把一个仓库的日常协作拆成多个小能力：
  - 认识仓库当前状态
  - 接手或切换分支
  - 本地增删改查文件
  - 提交并推送改动
  - 创建 PR 并说明变更
  - 读取 PR 反馈并提炼行动项
  - 根据反馈修订并更新 PR
  - 完成一次 PR 协作闭环

- [`specs/虚拟机镜像静态处理`](./specs/虚拟机镜像静态处理)
  面向已经拿到离线镜像文件的执行者，把识别镜像、提取文件、转换格式和 Linux 离线恢复访问拆成多个小能力：
  - 识别镜像类型并确认可操作路径
  - 离线从虚拟机镜像提取文件
  - 转换虚拟机镜像格式
  - Linux 离线恢复登录访问

- [`specs/PPT内容生成与迭代`](./specs/PPT内容生成与迭代)
  面向写材料和做汇报的人，把根据输入生成或更新 PPT 内容拆成多个小能力：
  - 识别是新建整份还是局部迭代
  - 把输入材料整理成页面结构
  - 生成新页面或整份初稿
  - 更新已有PPT内容与版式
  - 导出并检查最终可交付版本

## 如何新增能力

- 如果你要解决的是一整条流程，优先先建一个场景目录，再把步骤拆成多个小能力
- 如果你要解决的是一个可以独立复用的动作，直接新增单个能力目录即可
- 如果你写的是强业务绑定的方法，明确写出适用边界，不要伪装成通用能力

## 约定（最小化）
- 不强制标准目录结构
- 只要是 Spec 且能解决真实问题，都可以收录
- 每个能力建议包含：
  - 至少一个 Spec 文件
  - 一份环境配置说明（`environment-setup.md`）

## 工具与安全策略
- 不在仓库内提供工具下载脚本
- 仅提供下载地址、版本建议与配置步骤
- 使用者需自行完成工具安装与环境准备

## 共建方式
如果你有独特想法或更优解，欢迎提交 Issue / PR，一起沉淀可复用、可执行、可迭代的 Spec。
