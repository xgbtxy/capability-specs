# capability-specs

把解决问题最有效的工具和方法提前整理好，让人和 AI 在执行中学习，而不是在搜索和试错里焦虑。

如果你对某个项目还不太了解，也没关系。遇到类似问题时，把项目丢给 AI，直接问它这里哪个能力可以解决。

这里的每一个 spec，基本都代表“某个工具可以解决的一类场景”。但场景一定是覆盖不完的，所以这个项目本来就适合一起慢慢补全、一起做好。

这个项目以后也不会只是一个人写，而是适合大家一起写。所以这里先把理念说清楚：

- 目录名优先写用户看得懂的场景，让人一看就安心
- 能力底座优先选非常强的开源工具，让 AI 真能落地
- 每个 spec 尽量写得简短、直接，像一个小能力包
- 相信大模型的能力，不需要把说明写成很厚的手册，只需要给它一个清晰结构

这个仓库很简单：

- 收集某件事能解决什么问题
- 收集处理它最好用的工具
- 收集最省事的方法

你也可以把这里理解成：

- 外面写场景
- 里面用强工具
- 给大模型一个结构
- 让它去完成问题

这样当人或 AI 进入一个目录时，可以很快知道：

- 这件事能不能做
- 用什么工具做
- 怎么把工具装起来
- 最小怎么开始

## 唯一硬要求

每个能力必须有一份 [environment-setup.md](./templates/environment-setup.md)。

这份文件只做一件事：

- 直接写清楚这个能力需要什么环境
- 直接给官方安装方法
- 直接给最小检查方式
