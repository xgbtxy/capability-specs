# tools-and-safety

## 目标

定义有效插件集合发布时的工具优先级、安全策略和风险控制边界。

---

## 1. 工具优先级

推荐的工具优先级如下：

1. Git + Git 凭据 + GitHub REST API
2. `gh`
3. GitHub MCP / 连接器
4. Playwright

原因：

- Git + REST API 最底层、最稳定
- `gh` 依赖安装环境
- MCP / 连接器依赖授权范围
- Playwright 依赖浏览器权限和页面稳定性

因此，本能力的默认实现路径是：

**Git + REST API**

---

## 2. 推荐工具组合

### 2.1 文件集合构建

推荐使用：

- PowerShell
- Python

用于：

- 读取 `vuln_registry.csv`
- 统计 `DONE`
- 生成 `include` 清单
- 检查正式文件是否存在

### 2.2 干净环境构建

推荐使用：

- `git worktree`

用于：

- 从 `origin/main` 派生干净发布环境
- 与脏工作树隔离

### 2.3 PR 创建

推荐使用：

- GitHub REST API

用于：

- 查询现有 PR
- 关闭旧 PR
- 创建新 PR

## 3. 安全策略

### 3.1 凭据策略

允许：

- 使用 `git credential fill` 获取本机 GitHub 可用凭据

禁止：

- 把 PAT / token / password 写入仓库
- 把凭据输出到日志文件
- 把凭据硬编码进脚本
- 提交任何包含密钥的配置文件

### 3.2 发布范围策略

允许：

- 只发布 `include` 清单中的文件

禁止：

- 从脏工作树整体复制文件
- 用当前目录快照替代 include 清单
- 把临时文档、说明文件、缓存文件带进 PR

### 3.3 PR 策略

允许：

- 关闭旧 PR 后重开新 PR
- 用统一标题和正文规范描述有效集合

禁止：

- 多个范围重叠的 PR 长期并存
- 在 PR 中混合有效结果和修复池
- 把未审计集合直接提为正式 PR

---

## 4. 常见失败点

### 4.1 GitHub 连接器 404

原因通常是：

- 连接器未安装到目标组织
- 连接器只对个人账号仓库可见
- 当前登录态不具备目标仓库权限

解决方法：

- 不依赖连接器
- 直接走 GitHub REST API

### 4.2 `gh` 不可用

原因通常是：

- 本机未安装 GitHub CLI
- PATH 未配置
- 登录态缺失

解决方法：

- 不阻塞发布
- 直接使用 `git credential fill + REST API`

### 4.3 脏工作树切分支失败

原因通常是：

- 当前仓库存在未提交修改
- 存在 untracked 文件会被 checkout 覆盖

解决方法：

- 不在主工作树切发布分支
- 直接使用新的干净 `worktree`

### 4.4 发布范围不清

原因通常是：

- 没有 include 清单
- 人工靠记忆挑文件
- 未以 `DONE + 文件存在` 为唯一准入标准

解决方法：

- 强制 include 清单驱动
- 强制台账+正式文件双条件校验

---

## 5. 最佳实践

1. 每次发布都重新生成摘要和完整性报告
2. 每次发布都重新生成 include 清单
3. 每次发布都从 `origin/main` 创建新的干净 worktree
4. 每次 PR 都明确写数量、范围、校验结果
5. 每次修订都在同一个发布 worktree 中继续完成
6. 每次发布后保留：
   - 分支名
   - commit
   - PR 编号
   - 统计结果

---

## 6. 结论

这个能力不是“帮你自动点网页发 PR”，而是：

**在最少依赖、最高可控的条件下，把有效插件集合稳定发布出去。**

因此它最重要的不是工具炫技，而是：

- 范围清晰
- 环境干净
- 凭据安全
- 结果可审计
