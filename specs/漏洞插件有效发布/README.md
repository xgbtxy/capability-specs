# 漏洞插件有效发布

把“脏工作树里的插件开发结果”收敛成“可审计、可复现、可提 PR 的有效发布集合”。

> 定位说明：这是一个特化发布能力，不是通用的仓库协作总能力。
> 如果目标是让不熟悉 Git 的执行者完成仓库查看、分支协作、提交推送、发起 PR、接收反馈并继续修订，优先参考 [`../GitHub仓库基础协作`](../GitHub仓库基础协作)。

## 这个能力解决什么问题

漏洞插件项目在持续迭代时，常见问题包括：

- 本地工作树很脏，混有历史实验、误删、回滚、临时修复
- 分支很多，难以判断这次 PR 到底该发什么
- 不是所有已写文件都应该进入正式发布
- `gh`、GitHub 连接器、网页自动化等能力并不总是可用
- 需要一种稳定的方法，把“当前有效插件集合”干净地发布到 GitHub

这个能力的核心目标是：

**从正式台账中筛出有效插件集合，复制到干净 worktree，提交并通过 GitHub REST API 创建 PR。**

## 适用场景

适用于以下情况：

- 插件仓库长期开发，工作树不干净
- 只想发布当前 `DONE` 且正式文件存在的插件
- 需要规避误发、夹带、幻觉残留
- `gh` 不可用，但本机 `git push` 正常
- 需要稳定、低依赖地创建 PR

## 关键原则

- 发布真相源只认：`vuln_registry.csv` + 正式文件真实存在
- 发布必须基于干净 `worktree`
- 发布范围必须通过 `include` 清单控制
- 不依赖 `gh`、GitHub MCP 或 Playwright
- PR 可直接通过 GitHub REST API 创建

## 最小使用方式

1. 根据台账生成有效插件集合
2. 生成 `include` 清单
3. 从 `origin/main` 起干净 `worktree`
4. 只复制 `include` 中的文件
5. `commit + push`
6. 用 `git credential fill` 取本机可用凭据
7. 调 GitHub REST API 创建 PR

## 输出结果

使用本能力后，应该得到：

- 一个干净发布分支
- 一次只包含有效插件集合的 Git 提交
- 一个明确的 GitHub PR
- 一份可解释、可复核的发布范围
