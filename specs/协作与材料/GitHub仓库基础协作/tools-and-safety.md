# tools-and-safety

## 目标

定义“新手 PR 协作流程”中的推荐工具路径、安全边界和常见误区。

---

## 1. 工具优先级

推荐顺序：

1. Git
2. GitHub 连接器 / GitHub REST API
3. `gh`
4. 浏览器手工操作

原因：

- Git 是最基础的真实状态来源
- 连接器 / API 适合读取 PR 信息和执行标准化操作
- `gh` 很方便，但依赖安装和登录状态
- 浏览器操作最容易遗漏上下文，不适合做默认路径

---

## 2. 安全策略

### 2.1 分支策略

允许：

- 在团队指定分支上继续工作
- 在原 PR 分支上继续修订

禁止：

- 未说明风险就切换到其他无关分支
- 被打回后默认重新开一个重复 PR

### 2.2 提交策略

允许：

- 只提交与当前任务相关的文件
- 使用清楚、可理解的提交说明

禁止：

- 把无关缓存、临时文件、测试产物一起提交
- 在不解释的情况下使用 `reset --hard` 等破坏性命令

### 2.3 反馈处理策略

允许：

- 先把 reviewer 意见整理成行动项再修改
- 修改后回复 reviewer 说明已处理范围

禁止：

- 没看懂评论就直接关闭问题
- 只回复“已修改”，却不说明改了什么

---

## 3. 常见失败点

### 3.1 改完了但没在正确分支

原因通常是：

- 开工前没确认当前分支
- 误在 `main` 或其他历史分支上改动

解决方法：

- 开工前先确认当前分支
- 再开始编辑文件

### 3.2 推送失败

原因通常是：

- 没有 push 权限
- 远端分支名写错
- 本机 GitHub 凭据失效

解决方法：

- 先确认远端仓库和目标分支
- 再检查当前账号权限和凭据状态

### 3.3 PR 反馈看不懂

原因通常是：

- 评论分散在多个 review / thread 中
- 建议、问题和必须修改项混在一起

解决方法：

- 先做反馈分类
- 再按“必改 / 建议 / 待确认”处理

---

## 4. 结论

这个场景最重要的不是记住很多命令，而是始终知道：

- 我现在在哪个分支
- 我准备提交哪些改动
- 我当前面对的是哪个 PR
- reviewer 到底希望我改什么
