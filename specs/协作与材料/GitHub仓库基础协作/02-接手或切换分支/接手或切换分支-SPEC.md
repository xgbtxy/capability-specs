# 接手或切换分支-SPEC

版本：v1.0  
状态：正式  
适用对象：不熟悉 Git 的执行者、第一次接手团队分支或需要切换分支的人

---

## 1. 问题定义

当别人说“你就在这个分支上改”时，新手经常不知道：

- 当前本地仓库是不是正确仓库
- 自己现在在哪个分支
- 目标分支是否已存在于本地
- 本地是否落后于远端
- 当前是否带着未处理改动，贸然切分支会不会出问题

因此，需要一个稳定的分支进入能力，用于：

**确认仓库、确认目标分支、同步状态，并进入可以安全修改的工作状态。**

---

## 2. 目标

1. 确认当前操作的是正确仓库
2. 确认目标工作分支
3. 尽量在不伤害本地改动的前提下切换到目标分支
4. 用自然语言告诉执行者现在能不能开始改

---

## 3. 标准流程

### 第 1 步：确认仓库和当前状态

至少要确认：

- 当前目录是 Git 仓库
- 当前分支名
- 是否存在未提交改动

### 第 2 步：确认目标分支来源

检查：

- 本地是否已有该分支
- 远端是否存在该分支
- 若本地不存在，是否需要创建跟踪分支

### 第 3 步：处理潜在风险

若当前工作树已有未处理改动，不能直接假装无事发生。必须：

- 明确告诉执行者当前有风险
- 优先避免覆盖或丢失本地改动

### 第 4 步：切换并同步

在安全前提下：

- 切换到目标分支
- 获取远端最新状态
- 说明本地是否已与远端同步

### 第 5 步：输出可理解结论

输出时应明确说明：

- 你现在在哪个分支
- 是否已是最新
- 是否可以开始改文件
- 如果还不能开始，卡在什么地方

---

## 4. 输出要求

最少应包含：

- 仓库路径
- 当前分支
- 目标分支
- 工作树是否干净
- 是否可安全开始修改

---

## 5. 安全边界

禁止：

- 未说明风险就覆盖本地改动
- 默认使用 `reset --hard`
- 把“本地有未处理改动”说成“没问题”

允许：

- 在解释风险后暂停切分支
- 在确认状态后帮助执行者继续进入下一步
