# 创建PR并说明变更-SPEC

版本：v1.0  
状态：正式  
适用对象：已把改动推送到远端分支、准备发起审核的新手执行者

---

## 1. 问题定义

执行者把代码推送上去之后，往往还不知道如何把它变成一个“可审核的 PR”。常见问题包括：

- 不理解 `base` 和 `head` 的关系
- 不知道标题该怎么写
- 不知道正文里应该放哪些信息
- 不知道怎样让 reviewer 快速看懂这次变更

因此，需要一个标准化的 PR 创建能力，用于：

**把远端分支上的改动转换成一个信息完整、边界清楚、方便审核的 PR。**

---

## 2. 目标

1. 正确指定 `base` 和 `head`
2. 生成清晰标题
3. 生成有信息量的 PR 正文
4. 返回明确的 PR 链接

---

## 3. 标准流程

### 第 1 步：确认分支关系

必须明确：

- `head` 是当前工作分支
- `base` 是目标合并分支

### 第 2 步：生成标题

标题至少应说明：

- 这次改动的核心对象
- 这次改动的主要动作

### 第 3 步：生成正文

正文建议包含：

- 改了什么
- 为什么改
- reviewer 应重点关注什么
- 是否做过验证

### 第 4 步：创建 PR

在分支关系和说明文案都已确认后，再创建 PR。

### 第 5 步：返回结果

返回：

- PR 编号
- PR 链接
- 后续等待 review 的说明

---

## 4. 输出要求

最少应包含：

- base 分支
- head 分支
- PR 标题
- PR 正文
- PR URL

---

## 5. 安全边界

禁止：

- 分支尚未推送就伪造 PR 已存在
- 使用无意义标题和空正文
- 把与本次改动无关的信息堆进正文

允许：

- 根据改动范围自动整理标题和正文
- 用 REST API、连接器或 `gh` 创建 PR
