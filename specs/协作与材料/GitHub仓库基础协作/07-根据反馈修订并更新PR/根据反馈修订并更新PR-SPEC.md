# 根据反馈修订并更新PR-SPEC

版本：v1.0  
状态：正式  
适用对象：已收到行动项、准备继续在同一个 PR 上完成修订的人

---

## 1. 问题定义

很多新手以为 PR 被打回后要“重新提一个新的 PR”，或者不知道是否还应该继续在原分支上工作。常见问题包括：

- 不知道应不应该继续用原分支
- 不知道改完后如何更新同一个 PR
- 不知道是否要回复 reviewer
- 不知道怎么说明哪些反馈已经处理

因此，需要一个稳定的修订能力，用于：

**在原工作分支上完成修订、推送更新，并让 reviewer 看懂这轮处理结果。**

---

## 2. 目标

1. 在原工作分支上完成修改
2. 生成新的提交并推送
3. 保持同一个 PR 持续更新
4. 清楚说明已处理项和未处理项

---

## 3. 标准流程

### 第 1 步：回到原分支

确认：

- 当前所在分支
- 该分支是否就是对应 PR 的 head 分支

### 第 2 步：执行修订

根据行动项逐条修订，不要在没处理完的情况下笼统说“已经全部改完”。

### 第 3 步：提交并推送

修订完成后：

- 创建新的提交
- 推送到原分支

原 PR 会因同一分支被推送而自动更新。

### 第 4 步：回复 reviewer

如有必要，应向 reviewer 说明：

- 已处理哪些问题
- 哪些问题暂未处理
- 若未处理，原因是什么

---

## 4. 输出要求

最少应包含：

- 已处理行动项
- 新提交说明或哈希
- PR 是否已自动更新
- 给 reviewer 的回复摘要

---

## 5. 安全边界

禁止：

- 默认新开重复 PR
- 不说明范围就回复“已修改”
- 把未处理项伪装成已解决

允许：

- 在同一分支上持续迭代
- 明确告知 reviewer 仍待确认的部分
