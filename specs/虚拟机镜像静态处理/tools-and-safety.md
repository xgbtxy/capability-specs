# tools-and-safety

## 目标

定义“虚拟机镜像静态处理”中的推荐工具路径、安全边界和常见误区。

---

## 1. 工具优先级

推荐顺序：

1. `qemu-img`
2. `guestfish`
3. `guestmount`
4. `qemu-nbd`

原因：

- `qemu-img` 适合先识别和转换镜像
- `guestfish` 最适合脚本化读取与修改
- `guestmount` 适合把镜像当文件系统来查看和复制
- `qemu-nbd` 更底层，适合需要块设备接入时使用

---

## 2. 安全策略

### 2.1 识别优先

允许：

- 先用只读方式识别镜像结构
- 先明确真实磁盘文件，再做后续操作

禁止：

- 看见 `ovf` 就当成磁盘直接处理
- 没确认格式就盲目转换或挂载

### 2.2 提取文件策略

允许：

- 优先只读提取所需文件
- 明确导出路径和目标文件范围

禁止：

- 不说明范围就全量复制整个镜像内容
- 在原镜像上直接做不必要写入

### 2.3 恢复访问策略

允许：

- 以“恢复登录访问”为目标离线修改 Linux 认证相关文件
- 修改后保留操作记录，便于回溯

禁止：

- 把能力写成提取原密码、凭证、密钥
- 把未验证的恢复方法直接当成通用结论

---

## 3. 常见失败点

### 3.1 把容器文件当成磁盘文件

例如：

- 把 `ovf` 当成可直接挂载的磁盘
- 忽略 `ova` 里真正的 `vmdk`

### 3.2 转换成功但后续不可读

常见原因：

- 原镜像本身损坏
- 多磁盘场景只转了一部分
- 转换参数不匹配

### 3.3 恢复访问后没有验证

常见问题：

- 虽然改了认证文件，但没有确认系统是否能正常启动
- 没有记录修改了哪些文件

---

## 4. 结论

这个场景最重要的不是一次把所有工具都用上，而是始终知道：

- 我手里的到底是什么镜像
- 我到底是在读文件、转格式，还是恢复访问
- 哪些步骤已经实战过，哪些还只是可执行设计
