# 提交并推送改动-SPEC

版本：v1.0  
状态：正式  
适用对象：已完成本地修改、准备上传改动的新手执行者

---

## 1. 问题定义

执行者在本地完成修改后，经常面临以下问题：

- 不知道哪些文件应该纳入本次提交
- 不知道提交说明怎么写得清楚
- 不知道应推送到哪个远端分支
- 害怕把无关文件或临时文件一起传上去

因此，需要一个稳定的提交与推送能力，用于：

**把当前任务相关的改动整理成清晰提交，并推送到正确的远端分支。**

---

## 2. 目标

1. 明确本次真正要提交的文件
2. 生成人类可理解的提交说明
3. 避免夹带无关文件
4. 把提交推送到正确远端分支

---

## 3. 标准流程

### 第 1 步：检查当前改动

需要检查：

- 已修改文件
- 新增文件
- 删除文件
- 明显无关的缓存、产物、临时文件

### 第 2 步：圈定本次提交范围

提交范围应围绕当前任务，而不是把“工作树里所有变化”全塞进一次提交。

### 第 3 步：生成提交说明

提交说明至少要体现：

- 改了什么
- 为什么改

不要使用无意义信息，例如：

- `update`
- `fix`
- `修改一下`

### 第 4 步：创建提交

只暂存本次需要进入提交的文件，然后生成提交。

### 第 5 步：推送到目标分支

推送前应确认：

- 当前分支
- 目标远端分支
- 当前账号是否具备 push 权限

---

## 4. 输出要求

最少应包含：

- 本次提交文件列表
- 提交说明
- 推送目标
- 推送是否成功

---

## 5. 安全边界

禁止：

- 默认 `git add .` 后不解释范围
- 把构建产物、缓存文件、无关文档一起提交
- 推送到与当前任务无关的分支

允许：

- 明确排除无关文件
- 先解释提交范围，再执行提交和推送
