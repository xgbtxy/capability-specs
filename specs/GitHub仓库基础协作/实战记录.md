### 2. 接手或切换分支

实际执行动作：

```powershell

```

实际结果：

- 成功切换到新分支 `codex/github-repo-basic-collaboration`
- 本地未提交改动被保留在新分支上

这一步的实战结论：

- 对当前这类文档整理任务，先切到工作分支再继续是合理路径
- 即使只是维护 Spec 仓库，也不应该长期直接在 `main` 上工作

### 3. 本地增删改查文件

实际执行动作包括：

- 新增 `specs/GitHub仓库基础协作/` 整组能力目录

- 修改仓库根 `README.md`
- 修改 `specs/漏洞插件有效发布/README.md`
- 目录从“新手PR协作流程”重构为“GitHub仓库基础协作”

这一步的实战结论：

- 在真实仓库里做文件增删改查时，必须始终围绕当前任务范围
- 目录重命名、文件新增和说明文档修改都应该被纳入同一任务语义里
- 在提交前，最好能说清楚自己到底动了哪些文件、为什么动

这一步的实战结论：

- 在明确提交范围后，文档型仓库也完全适合用正常分支协作流程
- `push` 成功与否是一个必须真实验证的点，不能只写“理论上可以”
