## 已跑通步骤

### 1. 认识仓库当前状态

实际检查结果：

- 当前本地仓库已存在，不需要重新 clone
- 当前远端为 `origin https://github.com/xgbtxy/capability-specs.git`
- 初始分支为 `main`
- `origin/main` 存在
- 工作树不是干净状态

当时工作树状态：

- `README.md` 已修改
- `specs/漏洞插件有效发布/README.md` 已修改
- `specs/GitHub仓库基础协作/` 为新增目录

这一步的实战结论：

- 在开始继续操作前，必须先识别当前仓库和工作树状态
- 如果不先看状态，很容易在 `main` 上继续裸改

### 2. 接手或切换分支

实际执行动作：

```powershell
git switch -c codex/github-repo-basic-collaboration
```

实际结果：

- 成功切换到新分支 `codex/github-repo-basic-collaboration`
- 本地未提交改动被保留在新分支上

这一步的实战结论：

- 对当前这类文档整理任务，先切到工作分支再继续是合理路径
- 即使只是维护 Spec 仓库，也不应该长期直接在 `main` 上工作

### 3. 本地增删改查文件

实际执行动作包括：

- 新增 `specs/GitHub仓库基础协作/` 整组能力目录
- 新增多个 `README.md`、`spec.yaml` 和正式 `SPEC.md`
- 修改仓库根 `README.md`
- 修改 `specs/漏洞插件有效发布/README.md`
- 目录从“新手PR协作流程”重构为“GitHub仓库基础协作”

这一步的实战结论：

- 在真实仓库里做文件增删改查时，必须始终围绕当前任务范围
- 目录重命名、文件新增和说明文档修改都应该被纳入同一任务语义里
- 在提交前，最好能说清楚自己到底动了哪些文件、为什么动

### 4. 提交并推送改动

实际执行动作：

```powershell
git add README.md "specs/漏洞插件有效发布/README.md" "specs/GitHub仓库基础协作"
git commit -m "docs: add github repo collaboration capability group"
git push -u origin codex/github-repo-basic-collaboration
```

实际结果：

- 提交成功
- 提交哈希：`7e02594`
- 远端分支创建成功：`origin/codex/github-repo-basic-collaboration`

这一步的实战结论：

- 在明确提交范围后，文档型仓库也完全适合用正常分支协作流程
- `push` 成功与否是一个必须真实验证的点，不能只写“理论上可以”

实际结果：

这一步的实战结论：

- 这组能力已经从“本地整理文档”走到了真实 GitHub 协作
- 到这一步为止，前 5 个能力都已经在真实仓库上被验证过

### 6. 读取PR反馈并提炼行动项

实际执行方式：

- 在 PR [#1](https://github.com/xgbtxy/capability-specs/pull/1) 中添加了一条可执行的“模拟 review”评论
- 评论内容不是泛泛而谈，而是明确提出 2 个动作：
  - 给新手补“最短起步路径”
  - 明确说明最后 3 个能力是通过“模拟 review + 同一 PR 修订”完成验证

提炼出的行动项：

1. 在 `README.md` 中增加“完全不懂 Git 时该怎么开始”的最短路径说明
2. 在本记录中写清楚最后 3 个能力的验证方式，避免误导读者

这一步的实战结论：

- “读取反馈”不一定非要等别人自然出现评论，只要反馈本身真实存在、且可执行，就可以验证这一步
- 关键不是评论来自谁，而是能否把评论准确提炼成动作

### 7. 根据反馈修订并更新PR

实际执行动作：

- 在同一个工作分支 `codex/github-repo-basic-collaboration` 上继续修改
- 根据模拟 review 补充 `README.md`
- 根据模拟 review 更新 `实战记录.md`
- 将修订后的内容提交并推送回同一个 PR

这一步的实战结论：

- 被指出问题后，不需要新开 PR
- 正确做法是在原分支继续修订，并把更新推送到同一个 PR

### 8. 完成一次PR协作闭环

本轮闭环方式：

- 起点：识别仓库状态并切换工作分支
- 中段：本地增删改查、提交、推送、创建 PR
- 收口：通过模拟 review 生成反馈，在同一 PR 上完成修订和再次推送

这一步的实战结论：

- 这组能力现在已经完成一次从“开始协作”到“收到反馈并完成修订”的闭环验证
- 虽然最后 3 步使用的是模拟 review，而不是外部 reviewer，但链路本身已经真实跑通
