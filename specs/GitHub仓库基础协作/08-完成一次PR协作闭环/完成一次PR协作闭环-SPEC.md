# 完成一次PR协作闭环-SPEC

版本：v1.0  
状态：正式  
适用对象：已经做完一轮仓库协作、需要确认当前是否闭环的新手执行者

---

## 1. 问题定义

仓库协作不是“发了一个 PR 就结束”，而是一个会反复往返的过程。执行者在中途常常会搞不清：

- 现在是在等 reviewer，还是自己还有动作没做
- 本地是否已经推送完
- PR 是否已经更新成功
- 这一轮是否已经走到可以暂停的状态

因此，需要一个闭环确认能力，用于：

**把一次 GitHub 仓库协作从起点到当前状态串起来，并明确当前阶段与下一步。**

---

## 2. 目标

1. 回看这轮协作已经做过哪些动作
2. 判断当前状态是否完整
3. 明确还有哪些步骤没完成
4. 给出下一步建议

---

## 3. 标准流程

### 第 1 步：确认当前事实

确认：

- 当前本地分支状态
- 当前 PR 状态
- 最近一次推送和更新情况

### 第 2 步：回顾已完成动作

至少检查：

- 是否已识别仓库状态
- 是否已切到正确分支
- 是否已完成本地文件处理
- 是否已提交并推送
- 是否已创建或更新 PR

### 第 3 步：判断当前阶段

阶段可能包括：

- 仍在本地修改
- 已推送但未创建 PR
- 已创建 PR，等待 review
- 已收到反馈，待修订
- 已修订并等待再次 review

### 第 4 步：输出结论

输出应明确：

- 当前在哪一步
- 哪些已经做完
- 哪些还没做
- 现在该继续做什么

---

## 4. 输出要求

最少应包含：

- 当前阶段
- 已完成步骤
- 未完成步骤
- 下一步建议

---

## 5. 安全边界

禁止：

- 忽略未处理反馈却声称流程已闭环
- 在状态不完整时给出“已经结束”的误导结论

允许：

- 明确指出当前应暂停等待 review
- 明确指出当前应返回上一能力继续处理
