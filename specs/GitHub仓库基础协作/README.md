## 为什么单独建这个文件夹

这个场景的目标不是教人背 Git 命令，而是让不熟悉 Git / GitHub 的执行者也能把一个仓库的日常协作玩明白。

如果直接把它写成一个大能力，例如“自动化处理 GitHub 仓库”，通常会有两个问题：

- 步骤太多，AI 容易在中间跳步或漏做检查
- 用户只卡在某一步时，能力颗粒度又太粗，难以复用

所以这里按实际协作过程拆成几个小能力，每个能力只解决一个明确问题。

## 适用对象

- 不熟悉 Git 命令的新手
- 第一次参与团队协作的人
- 想把一个仓库的日常操作逐步弄明白的人

## 包含的能力

- [`01-认识仓库当前状态`](./01-认识仓库当前状态)
  解决“这个仓库现在是什么情况，我该从哪里看起”
- [`02-接手或切换分支`](./02-接手或切换分支)
  解决“别人给了我一个分支，或者我要换到另一个分支，我怎么开始”
- [`03-本地增删改查文件`](./03-本地增删改查文件)
  解决“我在本地怎么安全地看、加、删、改文件”
- [`04-提交并推送改动`](./04-提交并推送改动)
  解决“我本地改完了，怎么安全上传”
- [`05-创建PR并说明变更`](./05-创建PR并说明变更)
  解决“我怎么把改动提给别人审核”
- [`06-读取PR反馈并提炼行动项`](./06-读取PR反馈并提炼行动项)
  解决“别人打回来了，我到底该改什么”
- [`07-根据反馈修订并更新PR`](./07-根据反馈修订并更新PR)
  解决“我改完后，怎么继续更新同一个 PR”
- [`08-完成一次PR协作闭环`](./08-完成一次PR协作闭环)
  解决“从开始到更新 PR，这一轮我是否已经走完整了”

## 建议使用顺序

1. 认识仓库当前状态
2. 接手或切换分支
3. 本地增删改查文件
4. 提交并推送改动
5. 创建 PR 并说明变更
6. 读取 PR 反馈并提炼行动项
7. 根据反馈修订并更新 PR
8. 完成一次 PR 协作闭环

## 最短起步路径

如果你完全不懂 Git，也可以先按下面这条最短路径开始：

1. 先看仓库当前状态，确认自己在哪个仓库、哪个分支
2. 切到团队让你工作的分支，不要直接在 `main` 上改
3. 在本地只改这次任务相关的文件
4. 提交并推送改动到远端分支
5. 创建 PR，把“改了什么、为什么改”写清楚
6. 收到反馈后，不要新开 PR，继续在原分支修改并推送

如果你只记住一句话，就是：

**先看状态，再切分支，再改文件，再提交推送，再在同一个 PR 上来回协作。**

## 设计原则

- 一个能力只覆盖一个关键动作
- 尽量少依赖高级工具，优先走稳定路径
- 所有输出都要让小白能直接看懂下一步该做什么
- 允许 AI 帮用户操作，但不假设用户理解底层命令

## 场景输出

执行完这组能力后，理想结果应该是：

- 用户知道仓库、分支、工作树当前处于什么状态
- 用户能在本地安全完成增删改查
- 用户能把本地改动安全提交到远端
- 用户能创建信息完整的 PR
- 用户能把 reviewer 反馈整理成待办项
- 用户能继续在同一个 PR 上完成修订和同步
